# Prior-Quality Audit Provenance

This note records the evidence chain for the prior-quality response audit so that Table 9 is inspectable rather than a prose-only explanation.

## Metric Source

`structure_fidelity` is emitted by the main pipeline before paper-facing audit tables are generated:

- metric call: `plan_b/pipeline.py`, where `query_structure = extract_structure(query.code)` and `structure_fidelity(query_structure.to_dict(), priors)` are stored in each episode result
- metric function: `plan_b/pipeline.py::structure_fidelity`
- raw result field: `structure_fidelity` in per-episode JSON rows under `results/mbpp224_fair_budget/*.json` and `results/prompt_only_structural_mbpp224_fair_budget/*.json`

The metric compares query reference-code structure with prior summaries through API-call, control-flow, and data-structure overlap. It is therefore retrospective and reference-dependent.

The field is visible in packaged raw result files before the paper-facing audit script runs. Example locations include:

- `results/mbpp224_fair_budget/mbppplus224_syntax_aware_multi_prior_mbrexec_budget8_seed1.json`, where `structure_fidelity` appears in episode rows, for example at line 264 in the current workspace snapshot.

## Bin Source

The main bins are:

- low: `< 0.15`
- medium: `0.15 <= x < 0.50`
- high: `>= 0.50`

These cut points match the earlier heuristic used in `scripts/54_train_usefulness_v2.py`, where low-fidelity examples use `fidelity < 0.15` and high-fidelity positive examples use `fidelity >= 0.5`.

## Audit Script

The paper-facing table is generated by:

- `scripts/67_make_prior_quality_audit.py`
- output markdown: `paper/tbl_prior_quality_audit.md`
- output JSON: `paper/tbl_prior_quality_audit.json`
- response alias markdown: `paper/tbl_prior_quality_response.md`
- response alias JSON: `paper/tbl_prior_quality_response.json`
- response-curve source: `paper/fig_prior_quality_response.md`
- convenience wrapper: `scripts/68_make_prior_quality_response.py`

The script reads stored raw-result fields. It does not:

- recompute retrieval
- rerun generation
- tune prompts
- tune retrieval
- filter episodes
- select only favorable conditions
- compute new `structure_fidelity` values for the paper table

The script computes quality coverage, bin-level net delta, harm rate, outcome-level fidelity separation, and threshold-sensitivity summaries from those stored fields. That aggregation is the paper table; it is not a new retrieval, generation, reranking, or fidelity-computation step.

## Reported Conditions

The audit reports:

- code-aware `syntax_aware` `single_prior`, `multi_prior`, `random_prior`, and `corrupted_prior`
- full `MBPP+224` prompt-only `prompt_structural` `single_prior` and `multi_prior`

This is intended to avoid cherry-picking only the favorable `multi_prior` diagnostic result.

## Interpretation Boundary

`structure_fidelity` is an audit variable, not a deployment variable. The paper uses it to ask whether observed outcome differences are consistent with prior quality after the run. It is not used to construct priors, select prompts, select candidates, filter tasks, tune retrieval, tune generation, or claim a deployable quality estimator.

## Sensitivity

`paper/tbl_prior_quality_audit.md` and `paper/tbl_prior_quality_response.md` report four forms of evidence:

- fixed low/medium/high bins with quality coverage, net delta, and harm rate
- outcome-level fidelity separation without binning
- threshold sensitivity for fidelity-qualified subsets at `0.10`, `0.15`, `0.25`, and `0.50`
- response-curve source rows for plotting net delta or harm rate by threshold

These analyses support diagnostic consistency. They do not turn `structure_fidelity` into a deployable quality estimator or a causal proof.

The complete threshold-sensitivity table is part of the paper-facing output for every reported setting, not a selected excerpt. The JSON copies in `paper/tbl_prior_quality_audit.json` and `paper/tbl_prior_quality_response.json` contain the same complete sensitivity records.
